# 3.6 AI辅助文档写作实战

产品经理的工作有相当一部分时间花在写文档上：PRD、方案对比、会议纪要、路线图......这些文档重要但耗时。AI不能替你思考产品方向，但它能大幅减少你在"把想法变成文字"这个环节上的时间。

这一节覆盖四个最常见的文档写作场景，每个都给具体的工作流和Prompt模板。

## 场景一：AI辅助写PRD

先说一个观点：AI写不了一份好的PRD。但AI能帮你把一份PRD的初稿写作时间从一天缩短到两个小时。剩下的时间你用来思考和打磨，最终产出的质量反而比以前更高。

### 工作流

**第一步：脑暴阶段。** 在动笔之前，先跟AI聊一轮。把你对这个需求的粗略想法说出来，让AI帮你梳理思路：

```
我在规划一个新功能，先跟你聊聊我的想法，帮我梳理一下思路。

背景：[简述背景和动机，比如：我们是一个B端协作工具，用户反馈最多的问题是"找不到之前的文件"。我们想做一个智能搜索功能来解决这个问题。]

我目前的初步想法是：[列出你想到的方案要点]

请帮我：
1. 补充我可能遗漏的关键问题（比如我没考虑到的用户场景、边界情况）
2. 对我的方案提出质疑——哪些假设可能是错的？
3. 建议这个功能应该包含哪些核心模块
```

这一步的目的不是让AI帮你做决策，而是**让AI当你的"第二大脑"**，帮你把思考漏洞暴露出来。

**第二步：生成PRD框架和初稿。** 脑暴完之后，你对这个功能应该有了比较清晰的想法。这时候让AI生成PRD：

```
请根据我们刚才讨论的内容，帮我撰写一份PRD文档。

## 文档信息
- 功能名称：[名称]
- 版本：V1.0
- 作者：[你的名字]
- 目标上线时间：[时间]

## PRD结构要求
请按以下结构输出：

### 1. 背景与目标
- 业务背景（为什么要做这个功能）
- 用户痛点（具体的用户问题描述）
- 功能目标（可衡量的成功指标）

### 2. 用户角色与场景
- 涉及的用户角色
- 每个角色的核心使用场景（用叙事方式描述，不要列表）

### 3. 功能需求详述
- 功能模块划分
- 每个模块的详细需求描述
- 输入/输出说明
- 异常处理逻辑

### 4. 交互说明
- 关键页面的交互流程（用文字描述，我会另外配原型图）
- 特殊交互说明

### 5. 数据需求
- 需要的数据埋点
- 关注的核心指标

### 6. 技术约束与依赖
- 已知的技术限制
- 外部依赖（比如第三方API）

### 7. 里程碑与排期建议
- MVP范围建议
- 分期迭代计划

请用专业但简洁的语言，避免冗余描述。每个需求点要足够具体，开发人员读完不需要再来问"这到底是什么意思"。
```

**第三步：审查与补充。** AI生成的初稿一定需要你仔细审查。重点看几个地方：业务逻辑对不对（AI不了解你公司的具体业务规则）、边界情况是否覆盖（AI容易漏掉一些只有你知道的特殊情况）、技术约束是否准确（最好拉技术同事一起看）。

对于需要修改的部分，直接告诉AI：

```
PRD的第3部分有几个地方需要调整：
1. [具体修改内容]
2. [具体修改内容]
另外，请补充以下场景的处理逻辑：[补充场景描述]
```

## 场景二：产品方案对比分析

产品经理经常需要做"方案A vs 方案B"的对比分析，给团队或管理层做决策参考。AI能帮你把对比做得更系统。

```
我需要做一个产品方案对比分析，请帮我进行系统评估。

## 决策背景
[描述你要解决的问题以及为什么有多个方案]

## 方案A
[描述方案A的核心思路和具体做法]

## 方案B
[描述方案B的核心思路和具体做法]

## 方案C（如有）
[描述方案C]

## 评估维度
请从以下维度进行对比（如果你认为还有我遗漏的重要维度，请补充）：
- 用户体验影响
- 开发工作量（估算为S/M/L/XL）
- 对现有架构的影响
- 可扩展性（未来迭代的灵活度）
- 风险与不确定性

## 输出要求
1. 用表格形式呈现各维度的对比
2. 每个维度给出简短的分析说明
3. 最后给出你的推荐方案及理由（但请明确标注这只是AI的初步建议，最终决策需要团队讨论）
```

拿到AI的分析后，你通常会发现它在"用户体验"和"可扩展性"这类主观维度上的判断未必准确，但在"结构化呈现对比信息"这件事上做得很好。把AI的输出当作一个思考框架来用，再填入你自己的判断，最终文档的质量会很高。

## 场景三：会议纪要与需求整理

开完需求评审会或用户沟通会，整理会议纪要是个苦活。特别是那种讨论很发散、各种想法飞来飞去的会议，事后你可能连谁说了什么都记不清了。

### 如果你有会议录音/转写

现在很多会议工具（腾讯会议、飞书、Teams等）都支持自动转写。拿到转写文本后：

```
以下是一段产品需求评审会的会议转写记录。请帮我整理成结构化的会议纪要。

## 会议信息
- 会议主题：[主题]
- 参会人：[列出主要参会人及其角色]
- 会议日期：[日期]

## 会议转写记录
---
[粘贴转写内容]
---

请按以下结构整理：

1. **会议结论**（最重要，放在最前面）：
   - 达成一致的决策
   - 确定的下一步行动（Action Item），包含负责人和截止时间

2. **讨论要点摘要**：
   - 按议题分类整理讨论内容
   - 每个议题标注是"已达成共识"还是"待进一步讨论"

3. **分歧与待决事项**：
   - 有哪些问题存在不同意见？各方观点分别是什么？
   - 建议的解决方式（比如需要谁来拍板、需要补充什么信息后再决策）

4. **提到但未展开讨论的话题**：
   - 可能需要另开会议讨论的事项

注意：会议转写可能有识别错误，请根据上下文合理修正明显的错误。如果某段话你无法确定含义，用[?]标注。
```

### 如果你只有自己的笔记

有时候会议没有录音，你只有自己匆匆记的笔记。这种情况下AI也能帮忙：

```
以下是我在一场用户沟通会中做的快速笔记（比较碎片化）。请帮我整理成一份完整的沟通纪要，并从中提取产品需求。

我的笔记：
---
[粘贴你的碎片化笔记]
---

请输出：
1. 沟通纪要（把我的碎片笔记整理成通顺的段落）
2. 从沟通中提取的用户需求列表（用用户故事格式）
3. 你觉得我的笔记中有哪些信息不够完整，建议我补充的
```

## 场景四：产品路线图草案

季度规划或年度规划时，产品经理需要输出一份产品路线图。这个过程AI能帮你整理思路和生成初稿。

```
请帮我生成一份产品路线图草案。

## 产品现状
- 产品名称：[名称]
- 当前版本核心功能：[列出主要功能模块]
- 当前面临的主要问题：[列出用户反馈最多的问题、业务上的挑战]

## 已确定的需求池
[列出你已经收集到的需求，如果有优先级判断也一并写出]

## 规划周期
[比如：2025 Q2-Q4，分三个季度]

## 业务目标
[比如：用户数增长50%、付费转化率提升到8%、NPS提升10分]

请输出：

1. **主题划分**：将需求池中的需求归类为3-5个产品主题（Theme），每个主题用一句话说明其战略意义

2. **季度规划建议**：
   - 每个季度建议聚焦的主题和具体功能
   - 功能排序的逻辑说明（为什么这个先做、那个后做）

3. **里程碑定义**：每个季度结束时，产品应该达到什么状态？用一句话描述

4. **风险提示**：这份规划中有哪些依赖或风险需要提前关注

请以适合团队讨论的草案形式输出，不要写得太definitive——路线图是用来讨论的，不是用来宣布的。
```

## 写文档时用AI的核心心法

用AI写文档这件事，我见过两种极端。一种是让AI生成完就直接交差，文档看起来工整但缺乏灵魂——读的人能感觉到这不是一个真正理解业务的人写的。另一种是完全不信任AI，每一个字都自己写，AI只用来检查错别字。

我建议的方式在中间：**让AI做结构化和初稿的工作，你来做判断和润色的工作**。具体来说，AI负责搭框架、填充标准化的内容、保证格式规范和逻辑完整；你负责填入业务判断、删掉正确但没用的废话、加入只有你知道的上下文信息。

这样做的好处是，你省下来的时间不是浪费了，而是花在了更有价值的思考上。一份好的PRD不是因为格式漂亮，而是因为背后的产品思考到位。AI帮你解放了格式的负担，你就有更多精力投入思考。
