# 3.2 AI产品设计方法论

上一节我们聊了思维层面的东西，这一节落到方法上。当你决定要在产品里加一个AI功能，从需求分析到交互设计到MVP定义，每一步跟传统功能设计都有不同。搞清楚这些差异，能帮你少走很多弯路。

## 需求分析：三步走，别跳步

传统功能的需求分析，路径通常是：用户有问题 → 想方案 → 评估可行性。AI功能的需求分析多了一个环节：**能力匹配**。

**第一步，定义用户问题。** 这一步跟以前一样，但要注意一个陷阱——别把"用AI"本身当作用户需求。"用户希望有AI功能"这不是需求，"用户希望快速找到历史订单中的异常数据"这才是。

**第二步，AI能力匹配。** 你得把用户问题翻译成AI的语言。"快速找到异常数据"对应的AI能力是什么？可能是自然语言查询（用户用大白话描述想找什么）、异常检测（AI自动标记不正常的数据点）、或者模式识别（AI发现用户没注意到的趋势）。不同的AI能力，实现难度和效果完全不同。

**第三步，可行性评估。** 这一步要跟技术团队一起做。核心要搞清楚四件事：数据够不够（AI需要什么数据，我们有没有）、效果能到什么程度（准确率大概多少，能否满足用户预期）、成本多少（API调用费、计算资源、开发周期）、有没有合规风险（用户数据的处理是否合规）。

这三步走下来，你会发现有些一开始很兴奋的想法在第二步就卡住了——AI目前做不到；有些在第三步被砍掉——技术上能做但成本太高。这很正常，筛选本身就是需求分析的价值。

## 交互设计三原则：透明、可控、容错

AI功能的交互设计，跟传统功能最大的区别在于：**输出结果不确定**。同样的输入，AI可能给出不同的结果；AI也可能犯错。这就要求交互设计围绕三个原则来做。

**透明性。** 用户要知道AI在做什么、基于什么做的。最差的体验是一个黑箱：用户点了一下，等了几秒，出来一个结果，不知道怎么来的。好的做法是告诉用户："根据你过去30天的使用数据，我建议..."或者"我分析了这3份文档，发现以下关键信息..."。透明不是要暴露技术细节，而是让用户能判断AI的结果是否可信。

**可控性。** 用户要能干预AI的行为。最基本的：能编辑AI的输出、能重新生成、能调整AI的参数（比如输出的详细程度、风格偏好）。进阶一点：能告诉AI"这个方向不对，我想要的是..."，让AI基于反馈调整。

**容错性。** 假设AI一定会出错，然后设计你的交互。AI生成的内容不要自动发布，先让用户确认；AI做的分类打标，要留一键修正的入口；AI推荐的操作，用户要能撤销。一个好用的撤销按钮，比把AI准确率从95%提升到97%更能提升用户满意度。

## AI功能的MVP：最小可用的AI体验

很多产品经理在规划AI功能时，一上来就想做一个全能的AI助手。这是大忌。

AI功能的MVP应该遵循一个原则：**选一个场景，做到"有用"就发**。

什么叫"有用"？用户用完之后觉得"嗯，这个确实帮我省了点事"就够了。不需要"哇塞这也太智能了"。

比如你想做一个AI辅助客服系统。MVP不是"AI自动回答用户所有问题"，而是"AI帮客服人员生成回复草稿，客服确认后发送"。这个版本AI出错没关系（客服会兜底），但已经能实实在在节省客服的打字时间。

MVP阶段还有一个很重要的目的：**收集真实使用数据**。AI功能的迭代非常依赖用户的实际使用反馈。用户什么时候用、怎么用、哪些AI输出被采纳了、哪些被修改了——这些数据比你在会议室里做的任何假设都值钱。

## 竞品AI功能分析：看什么

分析竞品的AI功能时，别光看宣传页面上写了什么，要实际去用。重点关注四个维度：

**功能范围。** 它的AI覆盖了哪些场景？是点状的功能（比如只有AI搜索），还是面状的（AI贯穿多个环节）？

**交互方式。** 用户怎么触发AI？是对话框、是按钮、是自动触发？AI的输出形式是什么？文本、列表、可视化？用户能不能给反馈？

**实际效果。** 找几个真实场景测试一下。准确率怎么样？响应速度如何？复杂问题能不能处理？边界情况（比如输入很模糊或很奇怪的内容）AI怎么应对？

**商业模式。** AI功能是免费的还是付费的？是基础功能还是高级功能？调用次数有没有限制？这些背后反映的是它的成本结构和定价策略。

## AI需求文档的特殊之处

写AI功能的PRD，跟传统PRD有几个关键差异：

第一，**输出不能写死**。传统PRD你可以说"点击按钮后，显示XX内容"。AI功能的PRD要这样写："点击按钮后，AI根据XX输入生成XX类型的内容，展示格式为XX。"你定义的是输入、输出类型和展示格式，不是具体输出内容。

第二，**要定义"可接受的失败"**。AI会出错，你的PRD要明确：什么程度的错误是可接受的？出错时用户看到什么？怎么让用户方便地纠正？

第三，**要写Prompt设计方向**。虽然最终的Prompt是技术团队调试的，但产品经理应该在PRD里说明AI的"人设"和"边界"——AI应该用什么语气？哪些话题AI可以回答、哪些应该拒绝？输出应该多长？这些直接影响用户体验，不能全扔给工程师。

第四，**要有评估标准**。传统功能上线后看的是bug数和用户反馈。AI功能还需要看准确率、采纳率（用户有多少比例接受了AI的输出）、和效率提升（用AI和不用AI的时间差）。这些指标要在PRD阶段就定好，否则上线后你不知道这个功能算成功还是失败。

方法论说到底就是一个框架，帮你在复杂的决策中不遗漏关键问题。但框架不能替代判断力，判断力来自实践。下一节我们聊具体的应用场景规划，看看这些方法论怎么在真实场景中落地。
